1Visión General

Prácticas 360 es una plataforma web integral para la gestión y seguimiento de prácticas preprofesionales de estudiantes universitarios. Su objetivo es centralizar el registro de actividades diarias, facilitar la supervisión docente y automatizar la generación de informes institucionales.

Funcionalidades Principales

  • Registro de actividades diarias: Los estudiantes registran fecha, horas, título y descripción técnica de cada actividad realizada.
  • Asistente IA: Generación automática de descripciones técnicas profesionales mediante Groq.
  • Supervisión docente: Los profesores pueden aprobar o rechazar informes con retroalimentación.
  • Dashboard analítico: Visualización de horas aprobadas, distribución de estados y semáforo de riesgo por estudiante.
  • Generación de informes Word: Exportación del informe institucional en formato .docx con plantilla oficial.
  • Gestión de perfiles: Cada usuario mantiene su perfil con datos institucionales y de práctica.

2Arquitectura del Sistema

El sistema sigue una arquitectura cliente-servidor con un frontend SPA en React que se comunica tanto directamente con Supabase (CRUD de informes) como con un backend Express (autenticación JWT e IA).

Diagrama de Arquitectura

Flujo de Comunicación

  1. Frontend → Supabase: Operaciones CRUD de informes diarios, perfiles y autenticación mediante SDK de Supabase.
  2. Frontend → Backend (proxy): Peticiones a /api/* redirigidas por Vite al backend Express.
  3. Backend → Supabase: Consultas a base de datos mediante Prisma ORM para autenticación.
  4. Backend → Groq: Peticiones a la API de Groq para generación de descripciones con IA.

3.5 Funcionalidades Clave

✨ Mejorar con IA (Groq)

Botón ubicado dentro del textarea de descripción, en la esquina inferior derecha. El flujo asíncrono sigue un patrón Request/Response controlado por el estado cargandoIA (booleano React). Al hacer clic:

  1. Validación previa: si tituloActividad.trim() === '', el botón permanece deshabilitado (disabled={cargandoIA || !tituloActividad.trim()}).
  2. setCargandoIA(true) bloquea la interfaz: el botón muestra "Mejorando...", se deshabilita el textarea y se previenen condiciones de carrera (Race Conditions) al impedir un segundo clic mientras la petición esté en curso.
  3. Envía POST /api/ai/enhance-description con payload estructurado { title: tituloActividad, currentDescription: descripcionActividad } mediante fetch() asíncrono.
  4. El backend construye un prompt profesional y consulta Groq (modelo llama-3.3-70b-versatile).
  5. La respuesta reemplaza automáticamente el contenido del textarea mediante setDescripcionActividad(data.description).
  6. Finalmente setCargandoIA(false) restaura la interactividad del formulario.

Este mecanismo de latch booleano (true/false) sobre cargandoIA mitiga condiciones de carrera: cualquier petición previa que aún no haya resuelto queda huérfana (el estado loading impide su procesamiento) y el usuario recibe siempre la descripción de la última llamada exitosa.

📄 Generación de Informe Word

Usa docxtemplater + PizZip + file-saver. La plantilla oficial debe estar en public/plantilla_institucional.docx. El flujo a bajo nivel opera así:

  1. Descompresión: fetch('/plantilla_institucional.docx') obtiene el binario como ArrayBuffer. new PizZip(arrayBuffer) descomprime el formato OPC (Open Packaging Convention) del .docx en memoria, exponiendo el árbol XML de partes (document.xml, styles.xml, etc.).
  2. Parseo XML: new Docxtemplater(zip) recorre el DOM de document.xml buscando tokens estructurales como {FACULTAD}, {CARRERA}, {NOMBRE_ESTUDIANTE} y directivas de bucle {#informes}...{/informes} que delimitan bloques repetitivos.
  3. Inyección de datos: doc.render() recibe un objeto JSON con la carga útil estructurada. Para el bucle {#informes}, mapea el arreglo informesAprobados.map(i => ({ id, fecha, horas, actividad_completa })) y Docxtemplater replica el bloque XML interno por cada elemento del arreglo.
  4. Empaquetado: doc.getZip().generate({ type: 'blob' }) re-comprime el ZIP en memoria con el XML modificado y saveAs(blob, 'Informe_Final_Practicas_...docx') (FileSaver) dispara la descarga asíncrona al navegador.

📊 Dashboard del Profesor

Utiliza Recharts con componentes declarativos que procesan la matriz de datos reactivos para renderizar vectores de rendimiento estudiantil sin sobrecargar el hilo principal:

  • <ResponsiveContainer>: Componente contenedor que ajusta automáticamente las dimensiones del gráfico al ancho del panel contenedor mediante observación de ResizeObserver, evitando deformaciones en ventanas redimensionadas.
  • <BarChart>: Renderiza las horas aprobadas por estudiante en orden descendente. Los datos se procesan con analyticsData.studentsHoursArray donde cada entrada contiene { name, horasAprobadas }. Recharts calcula la escala del eje Y automáticamente desde el dominio de los datos.
  • <PieChart>: Distribución de estados: Aprobados Rechazados Pendientes. Recharts calcula los ángulos proporcionales mediante la relación ángulo = (valor / total) * 360° y aplica interpolación SVG para las transiciones de entrada.
  • Semáforo de riesgo: Determinado matemáticamente según la densidad de registros acumulados del estudiante: Peligro Crítico (Rojo) si registros < 4, Riesgo Moderado (Amarillo) si 4 ≤ registros ≤ 6, Bajo Riesgo (Verde) si registros > 6. Cada tarjeta ejecuta analyticsData.studentRecords[studentId] para contar el total de informes del estudiante.
  • Sanitización de horas: En el cómputo de horas acumuladas se aplica parseInt(horas, 10) || 0 para neutralizar valores inyectados vacíos, strings huérfanos o ataques que generen NaN. Este patrón convierte cualquier valor no numérico a 0, garantizando que la base de datos reciba un entero válido y evitando corrupción en las agregaciones del dashboard.

4.3 Autenticación

El sistema implementa un doble sistema de autenticación que combina Supabase Auth para el frontend y JWT propio para el backend.

Flujo de Inicio de Sesión

1. Frontend: supabase.auth.signInWithPassword({ email, password }) 2. Frontend: consulta tabla 'profiles' WHERE id = authData.user.id 3. ¿Rol = ALUMNO? ├── Sí → Consulta 'student_data' WHERE profile_id = profile.id │ Almacena en userLogged: { id, student_id, nombre, facultad, │ carrera, ciclo, empresa, area, tutorEmpresarial } │ fetchReportsForStudent(student.id) └── No → Almacena en userLogged: { id, nombre, email, rol: 'PROFESOR' } 4. setIsLoggedIn(true) → Renderiza vista según rol

Middleware JWT

  • Extracción: Lee header Authorization: Bearer <token>
  • Verificación: jwt.verify(token, JWT_SECRET) — payload: { id, role }
  • Inyección: req.user = { id, role } disponible en controladores
  • Expiración: Token configurado con expiresIn: '8h'
  • Errores: Sin token → 401, Token inválido/expirado → 403

Flujo Asíncrono Request / Response

La comunicación entre el frontend y Groq sigue un patrón request/response mediado por el backend Express. El frontend envía un payload JSON estructurado con { title: string, currentDescription: string } mediante POST /api/ai/enhance-description. El backend valida que title no esté vacío (responde 400 en caso contrario), construye el prompt del sistema con las variables inyectadas, y ejecuta openai.chat.completions.create() contra la API de Groq. La respuesta viaja de vuelta como { description: string }.

Para mitigar condiciones de carrera (Race Conditions) durante peticiones concurrentes, el frontend gestiona el estado booleano cargandoIA. Antes de disparar el fetch(), se ejecuta setCargandoIA(true), lo que deshabilita el botón de activación (disabled={cargandoIA}) y bloquea el textarea de descripción mediante readOnly={cargandoIA}. Cualquier clic adicional durante la carga es ignorado. Al resolverse la promesa (éxito o error), se ejecuta setCargandoIA(false) restaurando la interactividad. Este latch impide que una segunda petición sobreescriba la respuesta de una anterior, garantizando que solo la última llamada exitosa actualice el estado del formulario.

7.6 Ciclo de Vida Transaccional del Estado de los Reportes

El estado de cada reporte sigue un ciclo de vida transaccional estricto: PENDIENTE → APROBADO o RECHAZADO → CORREGIDO (retorno a PENDIENTE). Cuando un docente rechaza un informe, el sistema persiste el estado RECHAZADO y el campo feedback con la retroalimentación obligatoria. El alumno visualiza el motivo del rechazo y puede editar el reporte exclusivamente en este estado. Al guardar la corrección, el frontend ejecuta UPDATE daily_reports SET approval_status = 'PENDIENTE', feedback = NULL, operational_notes = ? WHERE id = ?. La mutación automática del campo feedback a NULL y la restauración del estado a PENDIENTE son críticas para evitar bloqueos lógicos en la cola de revisión del docente: si no se reseteara el estado, el profesor no podría volver a evaluar el informe corregido. Este diseño garantiza que cada corrección reinicie el flujo de aprobación desde cero, manteniendo la integridad referencial del historial de retroalimentación.

7.7 Aislamiento de Datos con Row-Level Security (RLS)

Las tablas de Supabase a las que el frontend accede directamente (profiles, student_data, daily_reports) están protegidas mediante Políticas de Seguridad de Filas (RLS) de PostgreSQL. RLS opera a nivel de fila evaluando predicados definidos en políticas antes de ejecutar cualquier operación DML. Por ejemplo, la política para daily_reports verifica que student_id = auth.uid() en operaciones SELECT e INSERT, impidiendo que un alumno consulte o altere registros de otro mediante llamadas directas a la API de Supabase REST. Para las operaciones UPDATE y DELETE, la política refuerza adicionalmente que el reporte pertenezca al usuario autenticado. Este aislamiento cliente-servidor es transparente para el SDK de Supabase, que inyecta automáticamente el token JWT de sesión en cada petición. Sin RLS, un estudiante malintencionado podría modificar el student_id en el payload JSON de una petición HTTP directa y adulterar la bitácora de terceros; RLS mitiga este vector de ataque a nivel de base de datos.

Migración: Agregar Faculty/Career a Profiles

Para que el nuevo flujo funcione, ejecuta esta migración en el SQL Editor de Supabase:

-- Agrega facultad y carrera a la tabla profiles ALTER TABLE profiles ADD COLUMN IF NOT EXISTS faculty TEXT; ALTER TABLE profiles ADD COLUMN IF NOT EXISTS career TEXT; -- Actualiza profesores existentes (reemplaza con sus datos reales) UPDATE profiles SET faculty = 'Facultad de Ingenierías', career = 'Tecnologías de la Información (TI)' WHERE role = 'PROFESOR';

7 Sin paginación Problemas de rendimiento con muchos registros Paginación en backend + frontend Baja
8 Claves API en repositorio Riesgo de seguridad Rotar claves, agregar .env.example, usar secrets en producción Alta

11.3 Escalabilidad

Para escalar la aplicación a más facultades y usuarios, se recomienda:

  • Separar el backend en microservicios (auth, reports, ai)
  • Implementar caché con Redis para consultas frecuentes (dashboard, lista de estudiantes)
  • Migrar a una base de datos dedicada con mayor capacidad (fuera de Supabase Free Tier)
  • Implementar rate limiting en endpoints públicos
  • Agregar logging centralizado (ELK Stack o similar)
  • Contenerizar con Docker para despliegues reproducibles
